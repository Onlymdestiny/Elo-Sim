Anonymous submission

The problem

Ranked matchmakers pair players using hidden ratings (Elo, Glicko, TrueSkill). When those ratings misjudge skill, the result is a stomp: one team snowballs, the game ends in under 22 minutes, both sides leave unhappy.

User & need

  • User: a matchmaking engineer at a live game studio
  • Need: a pre-match predictor — takes only what the server has at lobby time (10 rating vectors) and returns a fairness score + Close / Competitive / Stomp label
  • Anti-requirement: no post-match features (gold graph, KDA). Those leak the outcome and are unusable at lobby time.

Track-2 contribution

  1. End-to-end prototype — Riot API collection → tier+LP MMR reconstruction → feature extraction → pluggable ML layer → real-data held-out evaluation.
  2. Robust data pipeline — Riot API collector with a dual-window token-bucket limiter and tier-balanced BFS seeding.
  3. Honest negative finding — all three classifier families converge to ≈50%, diagnosing a feature / label ceiling, not a model ceiling. Guidance a team can act on.

Grading-wise: Track 2 rewards system design + user focus + honest evaluation — not leaderboard accuracy.

Feature vector & label

Baseline (8-D)

  • μA, μB
  • Δμ (MMR gap)
  • varA, varB
  • σA, σB
  • pwin (TrueSkill closed-form)

Extended (14-D)

  • + avg winrate A, B
  • + winrate gap
  • + hot_streak / fresh_blood / inactive diffs

Label (3-signal vote)

SignalStompClose
Duration< 22 min> 35 min
Gold diff> 10k< 4k
Kill diff> 15< 7

Final label = worst of three votes. Forces all three axes to be close before calling a match Close.

Data retrieval pipeline

Tier-balanced seed sampling

Single-seed BFS collapses to one MMR cluster. We sample ~100 PUUIDs from every tier (Iron II → Diamond II plus Master / GM / Challenger apex rosters) and interleave before BFS expansion.

Dual-window token bucket

Riot dev keys: 20 req/s AND 100 req/2 min. A single-window limiter either wastes one cap or violates the other under burst. Two deques, sleep only as long as the more restrictive window requires → true 10-thread burst.

MMR reconstruction

Iron–Diamond: base[tier] + offset[div] + LP. Apex: 3200 + LP (continuous ladder). σ estimated from tier + games played + winrate deviation.

Throughput

500 matches in ~18 minutes; avg 2.2 s/match.

Why 50%? Feature ceiling, not model ceiling

Three families, one result

  • RF — high-variance bagging
  • XGB — sequential boosting
  • LogReg — linear / low-variance

Different points on the bias-variance curve. If the ceiling were model capacity, at least one would separate. They don't → signal is not in φ.

Where the signal went

  1. Label noise. Duration / gold / kill thresholds are defensible but not canonical; two reasonable raters would disagree on borderline games.
  2. MMR reconstruction loss. Tier + LP is a lossy projection of Riot's hidden matchmaking MMR. Two Diamond II 50 LP accounts can differ by hundreds of internal points — we can't tell them apart.

Why extended features hurt

  1. Collinearity with MMR. Per-team winrate correlates with tier — which MMR already encodes. New columns are near-redundant with existing ones.
  2. Sparsity at n=500. hot_streak / fresh_blood / inactive are rare per-player booleans. Tree splitters happily carve spurious leaves on noise.
  3. Curse of dimensionality. 8-D → 14-D with 400 training rows = ~75% fewer samples per dim. High-capacity models (RF, XGB) lose most; L2-regularised LogReg holds up (stays flat).

Adding features without new signal is a regression — this is the diagnostic finding.

What the engineer should take away

❌ Do not

  • Treat the score as a 90%-oracle
  • Use it as a hard gate for pairing
  • Ship the 14-D extended model

✅ Do

  • Use it as a soft fairness regulariser alongside existing skill-difference loss
  • Retrain with internal MMR if available — that closes most of the gap
  • Replace heuristic labels with studio-canonical ones

The prototype's value is not its accuracy — it's the diagnosis of where accuracy is bounded and what would lift it.

Reproducibility

# 1. Collect two independently-seeded 500-match datasets
python collect_real_data.py --tier-seed --target 500; `
  if ($?) { python collect_real_data.py --tier-seed --target 500 }

# 2. Train on first, evaluate held-out on second
python validate_real_data.py `
  --train-csv real_matches.csv `
  --test-csv  real_matches2.csv `
  --model xgb --features baseline
  • Auto-incrementing filenames (real_matches.csv → real_matches2.csv → …) — multi-run safe by default.
  • All thresholds / MMR constants / hyperparameters are module-level — no values buried inside loops.
  • Persistent .seen_matches.json de-duplicates across runs.

Limitations & future work

  • Hidden MMR. Partner with a rights-holder for access to Riot's internal matchmaking MMR — eliminates the reconstruction loss.
  • Label source. Replace the three-signal vote with studio-canonical quality ratings (or post-match player survey scores if available).
  • Dataset size. 500 matches is enough to diagnose the feature ceiling but not to train champion-specific or role-aware variants. 10k+ matches would enable that.
  • Cross-patch drift. Currently all matches from one patch window. Pipeline supports cross-patch evaluation but we did not measure the drift effect.
